# WorldCup Predictor 真实预测智能体设计

**日期**: 2026-04-19  
**状态**: 待用户 Review

## 1. 目标与范围

### 1.1 目标

- 将当前基于本地规则生成的最小预测逻辑，升级为真实可联网的预测智能体。
- 保持现有 `POST /api/predict/{match_id}` 接口不变，避免破坏已接通的前后端链路。
- 让预测结果具备结构化输出、可追溯证据、可版本化存档和后续可评估性。
- 为后续赛后评估、命中率统计和前端成功率看板提供稳定输入。

### 1.2 非目标

- 本阶段不做多智能体协作。
- 本阶段不做自动批量预测。
- 本阶段不做赔率、盘口和博彩相关信号接入。
- 本阶段不做赛后评估与统计接口实现。
- 本阶段不做证据缓存和复杂调度优化。

## 2. 已确认决策

| 决策项 | 选择 | 说明 |
|---|---|---|
| 预测基础 | 数据库事实 + 预测时临时抓取公开信息 | 数据库作为底座，预测时补足赛前上下文 |
| 可用外部信号 | FIFA 排名、近 5 场战绩、历史交锋、伤停/停赛、比赛场地与时间、预计首发、主流媒体赛前分析 | 不接赔率类信号 |
| 预测结果形式 | 结构化预测 + 简短理由 | 兼顾可评估性和前端展示 |
| 信息获取控制 | 固定来源优先，证据不足时允许补充搜索 | 在可控性和效果之间取中间方案 |
| 智能体形态 | 单智能体直出 | 暂不拆成多阶段或多智能体 |
| 证据刷新策略 | 每次手动点击都重新抓取最新证据 | 不复用跨请求缓存 |

## 3. 总体方案

### 3.1 核心思路

- 保留现有 `POST /api/predict/{match_id}` 作为唯一预测入口。
- 后端先读取数据库中的比赛事实，作为预测输入底座。
- 单智能体在一次调用中完成检索、判断和结构化输出，但其可用信号、来源优先级和输出格式都由系统强约束。
- 每次预测都生成一个新的 `prediction_versions` 版本，不覆盖历史。
- `matches.prediction` 继续保留“最新预测摘要”角色，用于首页和列表页直接渲染。

### 3.2 设计原则

- 可评估：预测结果必须直接映射到后续评估维度。
- 可审计：每次预测要保留证据来源、预测时间和模型信息。
- 可替换：模型 provider、prompt 和输出校验要与业务入口解耦。
- 最小破坏：尽量不修改现有前端契约和已有 refresh 流水线。

## 4. 证据策略与可信度规则

### 4.1 证据分层

- `Level 1: 官方事实源`
  - FIFA 官方页面、官方赛程、官方球队信息
  - 用于确认比赛事实、球队身份、赛程时间、场地、赛事背景
- `Level 2: 高可信统计/资料源`
  - 用于 FIFA 排名、近期战绩、历史交锋、球队近况汇总
  - 用于形成预测的主体硬信号
- `Level 3: 主流媒体赛前报道`
  - 用于伤停、停赛、预计首发、赛前舆论判断
  - 属于软信号，不得伪装成官方事实
- `Level 4: 补充搜索结果`
  - 仅在前 3 层证据不足时启用
  - 只能补缺，不能推翻更高等级来源的明确事实

### 4.2 硬规则

1. 高等级来源优先，低等级来源不得覆盖已确认的高等级事实。
2. 来源冲突时，优先保留更新、更可信的来源，并在结果中记录不确定性。
3. 伤停、预计首发、媒体分析只能影响预测判断和置信度，不能作为确定事实入库。
4. 若关键证据缺失，允许继续预测，但必须降低 `confidence` 并记录原因。
5. 每条证据必须带来源名称、来源 URL、来源等级和使用说明。

## 5. 预测输入与输出合同

### 5.1 预测输入

后端提交给预测智能体的输入由两部分组成：

- `match_facts`
  - `match_id`
  - `date`
  - `time`
  - `stage`
  - `group_name`
  - `venue`
  - `home_team`
  - `away_team`
- `database_context`
  - 当前库内该比赛的已有事实
  - 已有历史预测摘要
  - 与评估相关的稳定字段摘要

### 5.2 结构化输出 Schema

预测智能体必须输出合法 JSON，核心字段如下：

- `predicted_score`
  - `home`
  - `away`
- `outcome_pick`
  - `home_win` / `draw` / `away_win`
- `home_goals_pick`
- `away_goals_pick`
- `total_goals_pick`
- `confidence`
  - `0-100`
- `reasoning_summary`
  - 适合前端直接展示的短文本
- `evidence_items`
  - 3-5 条
  - 每条包含：
    - `claim`
    - `source_name`
    - `source_url`
    - `source_level`
- `uncertainties`
  - 记录伤停未确认、预计首发不稳定、媒体口径冲突等不确定项
- `model_meta`
  - `provider`
  - `model_name`
  - `predicted_at`
- `input_snapshot`
  - 记录本次预测使用的关键比赛输入摘要

### 5.3 与评估域的对齐

本结构必须能直接支持后续以下评估维度：

- 胜平负命中
- 比分命中
- 主队进球数命中
- 客队进球数命中
- 总进球数命中

## 6. 预测工作流

1. 前端调用 `POST /api/predict/{match_id}`。
2. 后端校验比赛存在且尚未开赛。
3. 后端读取比赛事实，组装预测上下文。
4. 后端调用真实预测 provider。
5. 单智能体按“固定来源优先，证据不足再补充搜索”规则实时获取外部证据。
6. 单智能体返回严格结构化 JSON。
7. 后端执行 schema 校验、值域校验和业务校验。
8. 校验通过后写入新的 `prediction_versions`。
9. 同步更新 `matches.prediction` 为最新摘要。
10. API 返回新预测结果。

## 7. 失败处理与运行约束

### 7.1 硬失败

以下情况直接失败，不写入新版本：

- 比赛不存在
- 比赛已开赛
- 模型输出不是合法 JSON
- 缺少核心字段
- 分数、置信度或枚举值非法

### 7.2 软失败

以下情况允许继续生成预测，但必须降置信度并写入 `uncertainties`：

- 部分外部证据未抓到
- 不同媒体口径冲突
- 预计首发或伤停无法确认
- 近期战绩或交锋样本不足

### 7.3 并发与时延控制

- 同一场比赛同时只允许一个活跃预测请求。
- 若前一个预测请求尚未结束，再次点击应返回冲突状态或处理中状态。
- 单次预测请求应设置整体超时，目标区间为 `60-90` 秒。
- 本阶段不做跨请求证据缓存。

## 8. 数据库与 API 改造范围

### 8.1 数据库策略

- `prediction_versions` 表结构本阶段不扩列，完整预测结果继续写入 `prediction` JSON 字段。
- `matches.prediction` 继续保存“当前最新预测摘要”，不承担审计职责。
- 后续如需高频统计，可再把部分字段从 JSON 拆为显式列。

### 8.2 API 策略

- 保持 `POST /api/predict/{match_id}` 路径和返回外层结构不变：
  - `match_id`
  - `prediction`
  - `cached`
- `prediction` 内部结构升级为真实结构化 schema。
- 本阶段不新增预测详情和历史接口实现，但设计上预留：
  - `GET /api/matches/{match_id}/predictions`
  - `GET /api/predictions/{prediction_id}`

## 9. 后端模块拆分

建议在现有实现基础上扩展以下模块：

- `backend/services/predictions.py`
  - 预测入口服务
  - 负责比赛校验、版本号递增、落库和更新 `matches.prediction`
- `backend/llm/provider.py`
  - 定义统一预测模型接口
- `backend/llm/openrouter_prediction.py`
  - 真实预测智能体的 OpenRouter 实现
- `backend/services/prediction_context.py`
  - 从数据库组装比赛上下文
- `backend/services/prediction_schema.py`
  - 校验和标准化模型输出
- `backend/services/prediction_prompt.py`
  - 维护预测专用 prompt 和证据协议

## 10. 测试策略

### 10.1 单元测试

- 比赛未开赛才能生成预测
- 模型输出非法 JSON 时不落库
- 缺失核心字段时不落库
- 合法输出会生成新版本
- 每次预测版本号递增
- `matches.prediction` 会同步更新为最新摘要

### 10.2 集成测试

- 从数据库事实组装上下文并调用 provider
- 真实 provider 结果经 schema 校验后入库
- 同一场比赛多次手动预测会保留完整历史
- 并发预测请求的互斥行为正确

### 10.3 端到端验证

- 选择 1-2 场未开赛比赛
- 调用真实 `POST /api/predict/{match_id}`
- 确认结果已写入 `prediction_versions`
- 确认首页和 `GET /api/matches` 可读取最新预测摘要

## 11. 实施顺序

1. 定义预测输出 schema 和 provider 接口。
2. 先补预测失败与成功路径测试。
3. 实现 OpenRouter 真实预测 provider。
4. 重写 `backend/services/predictions.py`，替换当前 seeded-default 逻辑。
5. 做一次真实端到端预测验证。

## 12. 推荐落地方案

本阶段推荐方案为：

- 单智能体
- OpenRouter 作为真实 provider
- 强约束 prompt
- 严格结构化输出
- 保持现有预测 API 不变

这是当前复杂度、可控性和可扩展性之间最平衡的方案。它足以支撑下一阶段的赛后评估与统计闭环，同时不会把当前系统过早推入多智能体和复杂 orchestration。
